    I wanted to try my hand at coding a simple game.

    Tasks:

    • create UI layout
    • design framework for storing cards
    • render a deck of playing cards
    • code rules for card placement

    Creating the UI

    I chose to go with the standard Solitaire layout for pile placements as it is the most intuitive for players. I chose to start out with clicking as the method for placing cards as opposed to dragging. I saw clicking as a more natural approach, especially for laptop users equipted with only touchpads.

    Designing the Framework

    I needed a way to save the location of each card. I decided to have wach pile--the deck, drawn cards, discard piles, and playable piles--be it's own array.

    Piles containing face-up AND face-down cards

    My earliest problem was figuring out how to handle which cards were visible to the user. The very first test run showed only the top card of the pile, which is acceptable at the start of the game, but once a new card is added to the pile the previous cards would flip and become invisible to the user.

    The "card" element was given a boolean assigned to the card to indicate if the card had been "flipped" or not. This way cards that had already been played mearly needed to have their boolean value set to true in order to remain visible to the user. This also helped with cycling through the deck, as after going through the deck once all the cards must be flipped back and returned to the deck. The presense of the "flipped" value made it simple to reset the deck

    Drawn cards

    Another prominant issue was allowing for the drawn cards to be visible at all times. Cards are drawn in groups of 3, and to allow for 3 cards to be visible at once they were grouped into "sets" of 3 using a div. However, once the top card was played, only the remaining 2 cards were visible instead of what were now the top 3 cards of the drawn pile

    Solving this issue was challenging. I didn't play solitaire much myself to know what the pile should look like, and after some research found that a popular solution was to shit the top card from the previous set to the bottom of the current. But how?

    The solution was to check how many cards currently resided in the "drawn" pile and get the remainder when divided by 3. The first set should contain that many cards, while each set to offer simply contained the usual 3

    Coding the Rules

    While assigning the logic to the game, many changes needed to be made to the original "card" object

    • Check a card's value
    • Check a card's suit
    • Check a card's color

    Up until this point I had used strings to hold the "suit" value. Which was acceptable when all you needed was to know whether two suits matched or not. However, this method did not allow me to compare the "color" of two cards when it came to building piles. Instead I decided to change the "suit" value to an integer where odd numbers were red and even numbers were black. This way I could determine easily if a card had the same suit as another for adding to the discard pile, as well as if their colors were opposite for building piles
